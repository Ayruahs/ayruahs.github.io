Consistent Hashing
19 July '17

Consistent Hashing is a kind of hashing in which resizing requires only a fraction of the keys to be remapped as opposed to traditional hash tables which often cause the remapping of all or nearly all of the keys. The idea was first introduced in 1997, in an academic paper ( Consistent Hashing and Random Trees: Distributed Caching Protocols for Relieving Hot Spots on the World Wide Web by David Karger et al) at MIT for use in distributed caching. It has since then been used in many services like Amazon’s in-house DynamoDB database service.

The usefulness of consistent hashing becomes apparent in the context of distributed systems. It elegantly solves some of the problems that come with scale. A common way to achieve load balancing is to hash(key) % n where n is the number of servers. This works well only till the point one needs to add or remove servers. When such addition/removal occurs, n changes and all the keys need to be remapped. When working with very large databases distributed in thousands of servers, such remapping of keys using traditional hashing can be disastrous if 10-15% of the servers shut down due to any reason. This problem is solved with consistent hashing.

Consistent hashing maps keys to the same node, as far as possible. The consistent hashing algorithm involves hashing both the keys and nodes to an interval which has key hashes. If any node is removed, then the keys contained in its interval are mapped to the adjacent node and if any node is added, the keys contained in the newly defined interval are mapped to the new node.

This can give rise to unequally sized intervals since the mapping of nodes is random. This problem is solved using virtual nodes which correspond to the physical nodes. So, when a new node is added to the system, the virtual nodes ensure that the new node takes the load from many other physical nodes rather than splitting the load of just the adjacent physical node.

Consistent hashing ensures that:
     • All nodes receive roughly the same number of keys
     • When a new node is added or removed, only a fraction of the keys have to be moved to a new location.
